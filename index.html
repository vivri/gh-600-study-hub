1️⃣ Foundations of Agentic AI in GitHub

Define agentic AI in the SDLC source ↗
Assistant vs. agent — the core distinction
Assistant (reactive)Agent (goal-driven)
Responds to a prompt, returns control to the userInterprets a goal and decides intermediate steps
Suggests code, explains output, summarizes changesUses tools: GitHub API, CI workflows, repository write operations
Takes no repository actions — the user applies each step manuallyProduces durable artifacts: branches, commits, pull requests
Doesn't independently move work forwardIterates based on feedback: checks, reviews, scans
The "is it an agent?" checklist
  • Maintains a goal across multiple steps
  • Decides intermediate actions on its own
  • Uses tools
  • Creates or modifies durable artifacts (branch / commits / PR)
  • Iterates based on feedback signals
How agent behavior appears in GitHub
  • Branches & commits — what changed
  • Pull requests — what is proposed, why, and for review
  • Workflows & checks — what evidence exists
  • Review comments & approvals — what humans accepted or rejected
The agent does not replace the workflow — it enters the workflow as a participant. Typical flow: security alert → create branch (e.g. agent/bump-dep-2026-04-03) → update dependency + lockfile → open PR with summary and plan → wait for CI and review, revise as needed.
The agent lifecycle: plan → act → evaluate source ↗
It's a loop, not a linear sequence: agents repeatedly plan, act, and evaluate until the task meets defined success criteria — or the work is handed off to a human.
Plan
  • The agent interprets the goal and determines steps. In high-quality systems plans are structured, reviewable artifacts — not hidden internal state
  • Planning artifacts in GitHub: a structured plan in the PR description; a linked issue or checklist outlining scope and success criteria
  • A reviewable plan includes: scope (what will change), success criteria (how you'll know it worked), and a rollback or escalation path
Act
  • Execute in the repo: create a branch, change files and push commits, open/update a PR, respond to review feedback
  • Execution stays bounded: a specific repository, a branch, PR workflows — never uncontrolled direct changes to the default branch
Evaluate
  • Evaluation signals: workflow runs and status checks (build/test/lint), code review feedback, security signals (code scanning incl. SARIF uploads, secret scanning alerts, push protection, dependency alerts)
  • Rulesets / branch protection can require checks to pass before merge — turning evaluation into an enforceable gate, not an informal suggestion
  • Exam-critical principle: evaluation must be grounded in system signals, not in the agent's confidence
Quality bar — every phase visible
  • The plan is inspectable · the action is bounded to repository workflows · the evaluation uses objective signals. When any piece is missing, trust degrades
GitHub as system of record and control plane source ↗
System of record — the artifacts
  • Repositories and branches · commits and pull requests · issues and discussions (context & intent) · workflow runs and artifacts (evidence) · review history (decisions)
  • These artifacts do double duty: they support development AND make agent behavior inspectable after the fact
Control plane — the enforcement points (memorize this table)
GitHub controlWhat it enforcesWhy it matters for agents
Pull requestsChanges are proposed before mergingMakes agent work reviewable and discussable
Required reviewsHuman approval gatePrevents unreviewed merges; supports accountability
Required status checksCI evidence before mergingConverts evaluation into enforceable policy
CODEOWNERSReview routing by pathRight experts supervise high-impact changes
Rulesets / branch protectionCentralized branch policyPrevents unsafe merges; consistent guardrails
EnvironmentsApprovals for deployments / secretsControls sensitive execution and secret access
GitHub Actions belongs in the control plane
  • Least privilege: default workflow token permissions conservative (read-only where possible); grant higher permissions only to jobs that need them; environments + approvals for sensitive secrets/deployments
"What the agent can do" often reduces to "what the workflow token and tool credentials can do." Also remember: enforcement depends on configuration and admin permissions — the supervision model works everywhere, but enforcement requires the controls to be turned on.
Practical examples
  • Humans may need to explicitly "Approve and run workflows" on agent PRs — a built-in guardrail against privileged workflows running on untrusted changes
  • A job targeting an environment with required reviewers waits until approval is granted
  • CODEOWNERS on sensitive paths (.github/workflows/, infra/) auto-requests the owning team's review
Responsibilities, risks, anti-patterns, and traceability source ↗
Responsibility does not move with execution. Agents participate, but the accountable parties remain the people who defined the task, set permissions, chose/configured controls, and approved the change. The system can propose; humans decide what is accepted.
The four common anti-patterns (know these cold)
Risk / anti-patternWhat it looks likeMitigation with GitHub controls
Planless executionPR has a diff but no plan or rationaleRequire a plan section via PR template; require review before merge
Over-permissioned agentsWorkflows can write broadly, access secrets broadlyLeast-privilege GITHUB_TOKEN; environments with required reviewers; restrict who can trigger workflows
Hidden reasoningOnly the diff is visible — no assumptions, scope, or decision trailRequire a plan; link workflow runs; record decisions in PR comments
Blind trust in automation"CI passed, ship it" — checks only validate what they're designed to detectCombine checks with CODEOWNERS, required reviews, risk-based approvals
Traceability trail in GitHub
  • PRs and commit history · review comments and approvals · workflow runs and uploaded artifacts (test reports, logs) · code scanning uploads/alerts · secret scanning alerts and push protection events · organization audit log events
Minimum audit trail for an agent contribution
  1. A stated goal (issue link or PR description)
  2. An inspectable plan (PR plan section or file)
  3. A bounded changeset (branch and commits)
  4. Automated evidence (workflow run and artifacts)
  5. Human judgment (review and approval)
  6. A clear outcome (merge, revert, or escalation)
The contributor model for agent-generated work source ↗
Treat agent output as a contribution, not as categorically different. This avoids two opposite errors: excessive suspicion (rejecting work because "AI wrote it") and excessive trust (accepting work because automation produced it). Evaluate work by the standards of the workflow, not the novelty of the author.
Review rubric for agent PRs
  • Intent — clear goal and visible plan?
  • Scope — files changed aligned with the plan?
  • Evidence — required checks pass? logs/artifacts available?
  • Ownership — CODEOWNERS reviewed sensitive areas?
  • Policy — complies with rulesets / branch rules / environments?
  • Fallback — rollback or escalation clear for high-risk changes?
What a good agent contribution looks like
  • Understandable (clear goal and plan) · Bounded (scoped changeset, least privilege) · Reviewable (right owners, evidence present) · Policy-compliant · Reconstructable (audit trail supports post-hoc analysis)

2️⃣ Designing Agent Architecture and SDLC Integration

Map agent responsibilities to the SDLC source ↗
  • Don't treat an agent like a general-purpose developer roaming the whole SDLC — map it to specific stages where GitHub can enforce boundaries
  • Most teams start by scoping agents to implementation and validation, where PRs and workflows provide natural control points
SDLC stage → GitHub surface mapping
SDLC stageTypical agent responsibilityPrimary artifact
PlanningDraft scope, plan steps, define success criteriaIssues, PR descriptions/comments, Agents tab
ImplementationCreate branch, make changes, open/update PRBranch, commits, pull request
ValidationRun checks, attach artifacts, iterate on failuresWorkflow runs, checks, artifacts
DeploymentUsually restricted; approvals for sensitive actionsEnvironments and deployment approvals
Architectural boundaries
  • Scope early to reduce blast radius: limit which directories an agent can modify by policy and ownership
  • Treat workflow and infra changes as higher risk than application code
  • Prefer PR-based work even for automation — avoid direct-to-default-branch changes
The canonical design boundary: "agents propose; humans and policy accept." A remediation agent's scoped job ends when the PR (with plan + expected success signals) is open — validation and acceptance happen through checks, reviews, and policy.
Inputs, outputs, and success criteria (task contracts) source ↗
The task contract
  • Inputs — what the agent needs: issue context, constraints, boundaries
  • Outputs — what the agent produces: plan + PR + evidence
  • Success criteria — how results are evaluated: checks, scans, review outcomes
Under-specified tasks produce changes that look plausible but don't solve the underlying problem — e.g., a dependency update that leaves the vulnerable version reachable transitively.
Example: vulnerability-remediation contract
  • Inputs: security alert/issue link; repo scope (changes allowed under src/ and dependency files, NOT infra/); constraints (no workflow changes without platform review, no secrets introduced, no direct-to-main pushes)
  • Outputs: PR with structured plan, bounded changeset on an agent branch, links to workflow-run evidence
  • Success criteria: required checks pass; the security signal is actually resolved; scope matches intent (no unexpected files); rollback/escalation path recorded for higher-risk changes
Key exam principle
  • "CI passed" is necessary but not sufficient. Success criteria must reflect real intent — "vulnerability resolved," not just "tests passed"
  • Enforce success as a system guarantee: make the validation workflow (tests + CodeQL analysis) a required status check via rulesets/branch protection so the PR cannot merge until it passes
Separate planning, reasoning, and execution source ↗
  • Planning = what will be done and why · Execution = concrete repo changes · Validation = evidence the outcome meets criteria
  • When mixed together, reviewers see only the final diff — losing early intent validation, quick misunderstanding detection, and scope control before impact
Option A: Plan-first pull request
  • Agent opens a PR containing only the plan (no code); reviewers discuss/refine/approve; agent then implements in follow-up commits or a new PR
  • Human validation happens before any code is written — minimizes early exposure
  • Use for: high-risk or hard-to-reverse changes (workflows, infra, auth, production); when alignment on intent is critical
Option B: Plan + execution in one PR
  • PR includes both structured plan (description) and initial commits; validation still required before merge
  • Risk is introduced earlier (during proposal), increasing reviewer effort — but GitHub enforcement still prevents unsafe merges
  • Use for: low/medium-risk, easily reversible changes where speed matters
Enforcing the boundary (4 mechanisms)
  1. Capability boundary — planning agents get read-only tools so they can't modify files
  2. Explicit handoff — execution only after plan approval, via deliberate transition to an implementation agent
  3. Tool gating in orchestrators — planning runs without tool execution; tools enabled only after plan acceptance
  4. "Plan mode" workflows — generate a plan artifact and pause before changes
Exam-critical: "instructions not to edit" are guidance; tool allowlists and gates are enforcement. The design choice isn't whether work is reviewed (it always is) — it's when code generation is allowed relative to human validation.
PR governance: templates, checks, CODEOWNERS, rules, environment gates source ↗
Pull requests aren't just collaboration tools — they are enforcement mechanisms and the primary control point for agent execution.
The safe workflow chain

Agent creates branch → opens PR (includes plan) → required reviews validate approach → Actions run required checks → all checks pass + approvals complete → PR can merge

Control 1 — PR template requiring a structured plan
  • .github/pull_request_template.md with required sections: Goal, Scope (paths/files), Steps, verifiable Success criteria, Risks + mitigations, Rollback/escalation plan, Evidence (workflow runs, scan results), Review checklist
Control 2 — "Plan Gate" required status check
  • A workflow job fails if the plan artifact is missing; an admin marks it as a required status check in rulesets/branch protection — turning "include a plan" from a process expectation into a system guarantee
Control 3 — CODEOWNERS routing
  • Example: /security/ @security-team · /.github/workflows/ @platform-team · /infra/ @platform-team · * @core-team
  • Combined with required review policies, high-risk paths can't merge without the right experts
If an agent can bypass required checks or merge without reviews, that's a workflow design failure, not a model problem.
Reliable workflows: autonomy levels, outputs, contexts, triggers source ↗
Risk-based autonomy model (memorize)
Task typeExample pathsAutonomy design
Lowdocs/, formattingAuto-merge after required checks (and reviews if configured) pass
Mediumsrc/, dependency bumpsPR required + checks + at least one review
Highinfra/, .github/workflows/CODEOWNERS + multiple reviews + stricter rulesets
CriticalProduction deploys, secretsEnvironment approvals — agent prepares but can't execute
Outputs are workflow contracts
  • Step outputs (echo "plan=..." >> "$GITHUB_OUTPUT") pass values between steps in the same job
  • Job outputs + needs: pass values across jobs (${{ needs.plan.outputs.plan }})
  • env vars configure runtime behavior — don't use them as a substitute for structured outputs
Context selection
  • github.* → event metadata and runtime decisions ("what triggered this run?")
  • vars.* → centrally managed, reusable configuration
  • env.* → job-level environment variables / runtime configuration
Safe triggering and defensive gating
  • A "PR-only" workflow may still be triggered by push, workflow_dispatch, or schedule — add a job-level condition: if: github.event_name == 'pull_request'
  • Environment approvals: a job with environment: production pauses until required reviewers approve — the agent can prepare, not execute
Operate agents: observability, tools, MCP, secrets, hooks, reliability source ↗
Minimum observability set (every agent task should produce)
  • A structured plan (PR description/discussion) · bounded PR + commit history · workflow-run links for required checks · uploaded durable artifacts (logs/reports) · review outcomes
  • Evidence must be traceable to a specific workflow run and specific commit ("which run produced this artifact, against what code state?")
  • Share evidence across jobs with upload-artifact/download-artifact; link runs in an "Evidence" section of the PR
Agent profile = three boundaries
  • Capability — which tools are allowed (prefer allowlists)
  • Visibility — whether the agent is user-selectable in interactive UI
  • Delegation — which subagents can be invoked and how handoffs occur
  • Read-only toolsets for planning/review agents; implementation tools restricted to execution agents; changes to tool allowlists are governance-sensitive
MCP and secrets hygiene
  • MCP servers can be remote endpoints or local processes; auth tokens injected at runtime via protected secret boundaries; prefer narrow tool subsets over wildcards
  • Adding/expanding MCP tools increases blast radius — review like a high-risk dependency
  • Never put secrets in instruction files, committed config, or plain workflow YAML; scope secret availability by environment
  • The agent's runtime has its own secret boundary — it does not automatically inherit repository CI secrets
Hooks (.github/hooks/) — enforcement independent of model reasoning
  • Pre-action hooks validate or block unsafe actions before execution (e.g., pre-tool-use blocking a delete command)
  • Post-action hooks log tool usage/outputs/decisions for auditing
  • Error hooks capture failures and trigger escalation or alerting
  • Unlike instructions, hooks are always applied during execution
Reliability patterns
  • Bounded retries: agent may revise the branch and rerun after a failed check
  • Escalation policy: if the same required check fails twice, escalate to a human with what failed, what was attempted, what evidence exists, and the suggested next step (prevents infinite loops)
  • Rollback-safe: deploy from explicit refs (commit/tag), not "latest on a branch"
  • Least privilege: e.g. permissions: contents: read, pull-requests: write — read content, comment on PRs, nothing more

3️⃣ Tooling, MCP, and Agent Execution Environments

How agents interact with GitHub APIs and workflows source ↗
  • Agents work through the platform, not around it: APIs to read state and act (create branches/commits, open/update PRs, trigger workflows), Actions workflows as controlled runners
  • All API requests are authenticated — PATs, GitHub App tokens, or the workflow GITHUB_TOKEN — so every action is permission-controlled and auditable
  • Agents follow the same change sequence as developers: select base branch → create working branch → modify files → commit → open PR
  • Copilot cloud agent operates in a GitHub Actions-powered environment
Traditional vs. agentic workflows ("Continuous AI")
Traditional Actions workflowGitHub Agentic Workflow
Deterministic, YAML-defined; every step/trigger/condition explicitMarkdown describes the outcome; the agent decides how to achieve it within constraints
—Frontmatter defines the boundaries: triggers, permissions, tools, safe-outputs
Runs as authoredCompiled into a lock file that GitHub Actions executes
  • Best for open-ended-but-bounded tasks: triage, reporting, documentation maintenance, CI failure analysis, code improvement. They extend CI/CD pipelines, not replace them
Agent sessions are observable
  • Monitor progress via session log, see actions taken, provide feedback mid-run, review the final PR
MCP servers, registries, and allow lists source ↗
The three-piece governance model (memorize)
  • MCP server exposes tools · Registry makes servers discoverable and trustable · Allow list decides which servers are permitted
Local vs. remote servers
  • Remote (hosted, accessed over network) — recommended for most users; less setup; e.g. GitHub MCP server at https://api.githubcopilot.com/mcp/
  • Local (your machine, e.g. http://localhost:3000) — tighter config control, access to local resources, typically no external auth
  • GitHub Enterprise Server: local only. Enterprise Cloud with data residency: both local and remote
  • Add in VS Code: Copilot Chat → Agent mode → Tools icon → Configure tools → Add MCP server → type HTTP → URL → scope (workspace/all) → Authenticate → save
Custom MCP registries — requirements
  • Must follow the MCP registry v0.1 spec and expose: GET /v0.1/servers, GET /v0.1/servers/{name}/versions/latest, GET /v0.1/servers/{name}/versions/{version}
  • Must include CORS headers (Allow-Origin *, GET/OPTIONS, Authorization/Content-Type)
  • Options: fork/self-host the open-source MCP Registry, run it locally in Docker, build your own, or use Azure API Center as a managed registry (enable anonymous access; enter the base URL only, without /v0.1/servers)
Allow lists — configuration facts
  • Enforced at org or enterprise level, tied to the Copilot seat that governs the user
  • Enterprise path: enterprise page → AI controls → MCP; Org path: Settings → Copilot → Policies
  • Policy options: Allow all (any server, no restrictions) vs Registry only (only servers in the configured registry)
  • Registry-only gotcha: even local MCP servers must be listed in the registry, and server IDs must match exactly. Policy applies immediately to all developers
Execution context and boundaries source ↗
Execution context = repository + branch + workflow + permissions. It determines both visibility and capability.
Repository scope
  • Agents operate only within their repository; no access to others unless explicitly granted
  • Configure per-repo: Settings → Code & automation → Copilot → Cloud agent → enable
Custom agent scope (within the repo)
  • Files live at .github/agents/*.agent.md (e.g. security-reviewer.agent.md)
  • applyTo limits which files/directories the agent focuses on (globs like src/auth/**); tools defines allowed actions (e.g. read_file, search_files)
  • Custom agents control: file focus, allowed tools, behavior instructions. They do NOT control: branch creation, PR behavior, execution isolation — those are enforced by the execution context/system
Branch-based isolation
  • Agents never work directly on main: new branch from the selected base branch → changes → PR to base. You can pick a non-default base branch from the Agents page
Workflow & permission boundaries
  • Workflows are controlled execution containers: clean environment, repeatable, logged
  • Invoke an agent in CI: workflow file with explicit triggers (workflow_dispatch, schedule, events), minimal permissions:, a defined runner, COPILOT_GITHUB_TOKEN, then npx @github/copilot-cli -p "…" --no-ask-user (add --agent security-reviewer for a custom agent)
  • GITHUB_TOKEN permissions gate reading contents, writing code, creating PRs, secrets access, triggering workflows — always explicit and minimized
Agentic Workflow guardrails (defense in depth)
  • Read-only tokens by default · safe outputs (agent proposes, a separate gated step decides what's applied) · zero secrets in the agent process · sandboxed containerized execution · network isolation with allowlisted outbound access · threat detection scanning proposed outputs before any write
Agent execution limits and protections source ↗
Platform limits on the cloud agent
  • Access limited to the repository where it works; changes on a separate branch, never directly on the default branch
  • The agent prepares changes — the human decides whether to create the PR, review, request updates, or discard. It doesn't finalize on its own
  • Repo/org settings control which workflows are allowed, what actions can run, and what GITHUB_TOKEN may do
The five safe-execution patterns (Domain 2 exam objectives)
PatternImplementation
Error handlingFail fast (continue-on-error: false), meaningful error logs, prevent partial/inconsistent changes
RetriesRerun failed steps; retry loops with sleep for transient failures (network etc.)
RollbacksNaturally supported by branch isolation + PR review; also close/discard the PR, or revert commits if merged
EscalationRequired PR review, auto-assigned reviewers, workflow steps notifying maintainers — critical decisions stay human
TraceabilityWorkflow logs, commit history, PR discussions; clear commit messages; changes scoped to a branch
Environment protections & loop prevention
  • Environments can require approvals, restrict secrets, and control deployment targets before agent-driven jobs continue
  • Loop-prevention fact: actions performed with GITHUB_TOKEN generally do NOT trigger additional workflow runs; GitHub App tokens and PATs CAN — design carefully to avoid recursive automation loops
  • Sessions stay visible while running: monitor logs, inspect actions, send follow-up prompts
